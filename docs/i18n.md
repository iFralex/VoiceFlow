# i18n Guide

VoiceFlow uses [next-intl](https://next-intl.dev/) for internationalisation.
Italian (`it`) is the primary language; English (`en`) is available via the
user menu and persists in the `locale` cookie.

---

## Locale resolution

The active locale is read from the `locale` cookie set by
`src/actions/locale.ts`. It falls back to `it` when the cookie is absent.

```
User selects "English" in the user menu
→ setLocale('en')  [src/actions/locale.ts]  writes cookie
→ middleware.ts     reads cookie, sets x-locale header
→ request.ts        resolves locale for next-intl context
→ all useTranslations / getTranslations calls use the new locale
```

No URL prefixes are used. The locale is entirely cookie-based.

---

## Directory structure

```
src/i18n/
  locales/
    it.json   ← Italian (primary)
    en.json   ← English
  request.ts  ← next-intl getRequestConfig (reads locale from cookie)
  server.ts   ← re-exports: t (= getTranslations), getLocale, getMessages
src/middleware.ts  ← reads locale cookie, sets x-locale header
```

## Namespaces

| Namespace    | Purpose                                                  |
|--------------|----------------------------------------------------------|
| `common`     | Shell chrome, search, notifications, marketing copy      |
| `nav`        | Sidebar navigation labels                                |
| `auth`       | User menu, sign-in/out, language & theme labels          |
| `campaigns`  | Campaign list, status labels                             |
| `contacts`   | Contact list and form labels                             |
| `credit`     | Credit balance, packages, top-up                         |
| `settings`   | Organisation, members, billing, profile                  |
| `compliance` | GDPR, opt-out, RPO, data retention                       |

## Adding a new translation key

1. Add the key to **both** `src/i18n/locales/it.json` and
   `src/i18n/locales/en.json` under the appropriate namespace.

2. Use the key in your component:

   **Server component (async):**
   ```tsx
   import { t } from '@/i18n/server';

   export default async function MyPage() {
     const translate = await t('campaigns');
     return <h1>{translate('title')}</h1>;
   }
   ```

   **Client component:**
   ```tsx
   'use client';
   import { useTranslations } from 'next-intl';

   export function MyComponent() {
     const t = useTranslations('campaigns');
     return <p>{t('new_campaign')}</p>;
   }
   ```

3. **Every UI string must pass through translations — no inline strings
   outside JSX.** The only exceptions are:
   - Brand names (VoiceFlow, etc.) that are never translated
   - `data-testid` attributes (test infrastructure only)
   - Technical string constants not shown in the UI

## Plurals and interpolation

next-intl uses ICU message syntax for plurals and variable substitution.

**Variable interpolation:**
```json
// locales/common/it.json
"marketing_copyright": "© {year} VoiceFlow. Tutti i diritti riservati."
```
```tsx
t('marketing_copyright', { year: new Date().getFullYear() })
```

**Plurals** (ICU):
```json
"calls_count": "{count, plural, one {# chiamata} other {# chiamate}}"
```
```tsx
t('calls_count', { count: n })
```

## Updating baselines after intentional locale changes

When you add or rename translation keys, update both locale files, then run:

```bash
pnpm typecheck   # confirms no missing key type errors
pnpm test        # confirms all unit tests pass
```

If you modify strings that are snapshot-tested, update the snapshots:
```bash
pnpm exec vitest run --update-snapshot
```

## Testing i18n in unit tests

Unit tests run against a global mock of `next-intl` (see
`src/test/setup.ts`) that returns **Italian** strings by default. This means
existing tests that assert Italian text continue to pass without wrapping
components in `NextIntlClientProvider`.

To test a component with **English** strings, override the mock for a single
test:

```tsx
import { useTranslations } from 'next-intl';
import { vi } from 'vitest';

it('renders English labels', () => {
  vi.mocked(useTranslations).mockReturnValueOnce((key: string) => {
    const en: Record<string, string> = { campaigns: 'Campaigns', /* ... */ };
    return en[key] ?? key;
  });
  render(<Nav role="owner" />);
  expect(screen.getByText('Campaigns')).toBeTruthy();
});
```
